# 工作流程指南

**创建日期**：2026-01-17  
**目的**：提供从案例拆解到模型提炼再到洞察归纳的完整工作流程

---

## 🎯 工作流程概览

### 完整工作流（3个阶段）

```
阶段1：案例拆解（单案例）
  ↓
阶段2：模型提炼（多案例）
  ↓
阶段3：洞察归纳（跨案例）
```

### 三种工作模式

#### 模式A：单案例深度研究
```
单案例拆解（深度模式）
  ↓
提炼模型雏形
  ↓
后续用其他案例验证和修正
```

#### 模式B：多案例对比研究（推荐）
```
拆解2-3个相似案例
  ↓
提炼共性模型
  ↓
归纳跨案例洞察
```

#### 模式C：主题式研究
```
选定主题（如：订阅模式）
  ↓
拆解3-7个相关案例
  ↓
提炼模型
  ↓
归纳洞察
```

## 📋 阶段1：案例拆解

### 目标
系统化拆解单个商业案例，理解其决策逻辑、执行过程和成败原因。

### 使用文档
- `01-knowledge-base/methods/商业案例拆解指令.md`

### 三种模式选择

#### 模式1：快速拆解 ⚡（<30分钟）
**适用场景**：
- 第一次接触案例，先建骨架
- 需要快速了解案例概况
- 批量筛选案例

**输出内容**：
- 一句话总结
- 三个关键决策
- 一个核心机制/模型
- 通用启发/可迁移原则

#### 模式2：深度拆解 🎯（1-2小时）
**适用场景**：
- 重点案例，需要深入理解
- 需要理解决策与执行的细节
- 准备提炼模型

**输出内容**：
- 时空还原
- 问题诊断
- 决策拆解
- 执行拆解
- 模型提炼
- 成败分析
- 迁移应用

#### 模式3：跨案例对比 🔬（2-3小时）
**适用场景**：
- 对比2-3个相似案例
- 提炼共性与边界
- 准备归纳洞察

**输出内容**：
- 对比维度表格
- 模式识别（机制层）
- 可迁移原则（带边界）

### 操作步骤

1. **选择案例**
   - 从 `01-knowledge-base/cases/` 或 `04-case-library/` 选择
   - 或从公开资料收集案例信息

2. **选择拆解模式**
   - 根据时间和需求选择快速/深度/对比模式

3. **使用Claude拆解**
   - 复制 `商业案例拆解指令.md` 中的提示词
   - 输入案例信息
   - 获得结构化分析要点

4. **生成文档**
   - 使用 `文档生成规范.md` 中的案例文档模板
   - 保存到 `01-knowledge-base/cases/` 或 `04-case-library/`

5. **更新索引**
   - 更新 `04-case-library/case-index.md`（如适用）

## 🏗️ 阶段2：模型提炼

### 目标
从多个案例中提炼可复用的商业模型（机制、变量、边界、可证伪、最小验证）。

### 使用文档
- `01-knowledge-base/methods/模型提炼提示词.md`

### 三种工作模式

#### 模式A：单案例模型提炼
**适用场景**：
- 从单个重点案例中先抽机制雏形
- 后续用其他案例验证和修正边界

**操作步骤**：
1. 选择一个深度拆解的案例
2. 使用模型提炼提示词
3. 提炼机制雏形
4. 标注"待跨案例验证"

#### 模式B：多案例模型融合（推荐）
**适用场景**：
- 有2-5个相似案例
- 需要提炼共性机制并写清边界

**操作步骤**：
1. 选择3-5个最能代表机制的案例
2. 使用模型提炼提示词（模式B）
3. 提炼共性机制
4. 明确边界和失败模式

#### 模式C：模型对比验证
**适用场景**：
- 已有多个模型
- 需要做取舍、对比、验证

**操作步骤**：
1. 选择多个相关模型
2. 使用模型提炼提示词（模式C）
3. 对比验证
4. 决定保留/合并/淘汰

### 操作步骤

1. **案例聚类**
   - 从已拆解的案例中选择3-5个
   - 确保案例能代表要提炼的机制

2. **使用Claude提炼**
   - 复制 `模型提炼提示词.md` 中的提示词
   - 输入案例信息
   - 获得模型卡（Model Card）

3. **生成模型文档**
   - 使用 `文档生成规范.md` 中的模型文档模板
   - 保存到 `01-knowledge-base/methods/`

4. **更新索引**
   - 在相关案例中补充"模型标签/相关链接"
   - 更新 `01-knowledge-base/README.md`（如维护了模型列表）

## 💡 阶段3：洞察归纳

### 目标
从案例库中发现跨案例的深层规律（洞察），形成可验证、可迭代的洞察文档。

### 使用文档
- `01-knowledge-base/methods/洞察归纳提示词.md`

### 三种工作模式

#### 模式A：主题式洞察（快）
**适用场景**：
- 先形成洞察假设与验证计划
- 快速探索某个主题

**操作步骤**：
1. 选定主题（如：订阅模式、飞轮效应）
2. 使用洞察归纳提示词（模式A）
3. 形成洞察假设
4. 制定验证计划

#### 模式B：对比式规律提炼（推荐）
**适用场景**：
- 有成功/失败案例对比
- 或不同路径的案例对比
- 需要提炼机制句并写边界

**操作步骤**：
1. 选择3-7个支撑案例（成功/失败对比）
2. 使用洞察归纳提示词（模式B）
3. 提炼机制句
4. 明确适用边界

#### 模式C：趋势演化洞察
**适用场景**：
- 分析某个趋势的演化
- 需要识别阶段、驱动力、拐点

**操作步骤**：
1. 选择3-7个时间序列案例
2. 使用洞察归纳提示词（模式C）
3. 分析趋势演化
4. 识别可证伪点

### 操作步骤

1. **主题聚类**
   - 从案例库中选择3-7个支撑案例
   - 确保案例能支撑要归纳的洞察

2. **使用Claude归纳**
   - 复制 `洞察归纳提示词.md` 中的提示词
   - 输入案例信息
   - 获得洞察卡（Insight Card）

3. **生成洞察文档**
   - 使用 `文档生成规范.md` 中的洞察文档模板
   - 保存到 `01-knowledge-base/methods/`

4. **更新索引**
   - 在支撑案例中补充"洞察引用/相关链接"
   - 更新 `01-knowledge-base/README.md`（如维护了洞察列表）

## 🔄 完整工作流示例

### 示例：研究"订阅模式"

#### Step 1：案例拆解（2-3个案例）

**选择案例**：
- Netflix（从DVD到流媒体）
- Spotify（音乐流媒体）
- 其他订阅模式案例

**操作**：
- 使用深度拆解模式
- 拆解每个案例的决策逻辑和执行过程
- 生成案例文档

**产出**：
- 3个案例文档（Netflix、Spotify等）

#### Step 2：模型提炼

**操作**：
- 使用模型提炼提示词（模式B：多案例模型融合）
- 输入3个案例文档
- 提炼"订阅模式"的通用模型

**产出**：
- 模型文档：`订阅模式.md`
- 包含：核心机制、关键变量、适用边界、失败模式

#### Step 3：洞察归纳

**操作**：
- 使用洞察归纳提示词（模式B：对比式规律提炼）
- 输入3个案例 + 订阅模式模型
- 归纳跨案例洞察

**产出**：
- 洞察文档：`洞察01-订阅模式的三个关键变量.md`
- 包含：机制句、证据链、适用边界、可证伪点

## 📊 工作流时间估算

### 单案例深度研究
- 案例拆解：1-2小时
- 模型提炼：30分钟
- 洞察归纳：30分钟
- **总计**：2-3小时

### 多案例对比研究（推荐）
- 案例拆解（3个）：3-6小时
- 模型提炼：1小时
- 洞察归纳：1小时
- **总计**：5-8小时

### 主题式研究
- 案例拆解（5-7个）：5-14小时
- 模型提炼：1-2小时
- 洞察归纳：1-2小时
- **总计**：7-18小时

## ✅ 质量检查清单

### 案例拆解检查
- [ ] 不是课程复述，而是因果链与决策权衡
- [ ] 关键结论标注了（事实/推断/待验证）
- [ ] 至少写清一个机制（输入→过程→输出）
- [ ] 给出了边界（不适用条件）
- [ ] 给出了可证伪点
- [ ] 给出了最小验证动作（1-2周）

### 模型提炼检查
- [ ] 机制是因果链而非口号
- [ ] 变量可观测/可操作
- [ ] 边界清楚（不适用≥3）
- [ ] 有反例思考
- [ ] 可证伪点明确
- [ ] 最小验证动作可在1-2周完成

### 洞察归纳检查
- [ ] 洞察是机制句，不是正确废话
- [ ] 有证据链（≥3案例或明确补证据计划）
- [ ] 边界清楚（适用/不适用）
- [ ] 有反例思考
- [ ] 可证伪点明确
- [ ] 有最小验证动作（1-2周）

## 🎯 使用建议

### 新手入门
1. **从单案例开始**：选择一个感兴趣的案例，用深度拆解模式
2. **提炼模型雏形**：从单案例中提炼模型，标注"待验证"
3. **积累案例**：拆解更多相关案例
4. **验证和修正模型**：用新案例验证和修正模型边界
5. **归纳洞察**：当有3-5个案例后，归纳跨案例洞察

### 进阶使用
1. **主题式研究**：选定主题，系统拆解5-7个案例
2. **模型对比**：对比多个相关模型，做取舍和验证
3. **趋势分析**：分析某个趋势的演化，识别拐点

### 最佳实践
1. **质量>数量**：前10个案例要深度拆解，建立标准
2. **持续迭代**：模型和洞察要持续验证和修正
3. **记录过程**：记录拆解过程，便于复盘和改进
