## 一、核心升级逻辑

### 1.1 现状诊断

**当前的"三湘问道"v1.0**:
- ✅ 有完整的7步评估框架
- ✅ 有44万字理论知识库(DOC-S001~S048)
- ✅ 有11个增值功能
- ❌ **缺少真实案例库** (对标分析时缺乏素材)

**根本问题**: 
> 系统有"理论框架",但缺少"实战案例弹药库"。就像有了武器设计图,但没有弹药库存。

**三湘问道 2.0 = 评估引擎 + 情报库 + 自我进化系统**

```
评估引擎 (现有)
    ↓
需要 → 案例弹药 (新增)
    ↓
产生 → 新的评估案例 (反哺)
    ↓
    循环强化
```

**核心升级**:
1. **建立AI应用情报库** (cases/)
2. **建立失败模式分类** (taxonomy/failure_modes.yml)
3. **建立证据账本系统** (每个断言都要溯源)

### 2.4 证据账本操作规范

#### 2.4.1 证据分级标准 (A/B/C)

**A级证据** (官方/审计/可验证):
- SEC filing / 审计报告
- 公司官方公告 (融资/财报)
- 政府公开数据
- 创始人在正式场合的确认 (发布会/采访)
- 合同/协议扫描件

**B级证据** (媒体/分析报告):
- 知名媒体报道 (TechCrunch, WSJ, 36Kr...)
- 行业分析报告 (Gartner, CB Insights...)
- 多个来源交叉验证的信息
- 公司员工在LinkedIn的信息

**C级证据** (社交媒体/推测):
- 创始人Twitter/微博自述
- 用户评论/社区讨论
- 基于公开信息的推算
- 未经证实的传闻

#### 2.4.2 可信度判定标准 (高/中/低)

**可信度 = 证据等级 + 逻辑一致性 + 时效性**

| 证据等级 | 逻辑一致性 | 时效性 | 可信度 |
|---------|-----------|--------|--------|
| A | 与其他信息一致 | <3个月 | 高 |
| A | 与其他信息一致 | 3-12个月 | 中 |
| A | 存在矛盾 | 任意 | 中 |
| B | 与其他信息一致 | <3个月 | 中 |
| B | 存在矛盾 | 任意 | 低 |
| C | 任意 | 任意 | 低 |

**判定要点**:
1. 即使是A级证据,如果与其他信息矛盾,可信度也要降低
2. 时效性很重要 - 1年前的数据在快速变化的领域价值降低
3. 多个B级证据相互验证 > 单个A级证据

#### 2.4.3 可复查要求 (反幻觉核心)

**每条证据必须可复查**:
```markdown
❌ 错误示例:
| 断言 | 来源 | 证据等级 | 可信度 | 日期 |
| ARR $10M | 媒体报道 | B | 中 | 2025-01 |

问题: 3个月后你找不到当时的依据

✅ 正确示例:
| 断言 | 来源 | 链接/出处 | 证据等级 | 可信度 | 日期 | 备注 |
| ARR $10M | TechCrunch | https://tc.com/perplexity-funding | B | 中 | 2025-01-05 | 创始人访谈 |

好处: 3个月后可以复查,如果链接失效可以用archive.org
```

**链接/出处字段要求**:
- 优先: 完整URL
- 次选: 书名+页码 / 文档名+位置
- 最次: 详细描述 (如: "2025年1月5日创始人在XX大会的演讲")

**备注字段用途**:
- 补充说明 (如: "未经审计", "自述数据", "推算值")
- 标注特殊情况 (如: "与XX来源存在矛盾")
- 记录验证方法 (如: "交叉验证通过")

### 2.5 关键设计原则

#### 原则1: 成功+失败同口径
```yaml
# 模板: app_case_file.md 同时支持成功和失败
status: ["增长", "稳定", "停滞", "转型", "关停"]
tags: 
  - "success:xxx"  # 成功机制标签
  - "failure:yyy"  # 失败模式标签
  - "moat:zzz"     # 护城河标签
```

**为什么重要**: 
- 避免幸存者偏差
- 失败案例的学习价值往往更大
- 对标分析必须同时展示成功和失败

#### 原则2: 证据分级 (A/B/C)
```yaml
# evidence_levels.yml
A级: 官方数据/审计报告/创始人直接确认
B级: 媒体报道/行业分析报告
C级: 社交媒体/用户评论/推测

规则:
- 融资数据: 必须A级或B级
- ARR/用户数: unknown优于C级瞎编
- 失败原因: 优先A级访谈,其次B级报道
```

**为什么重要**:
- 避免编造数据
- 评估报告的可信度依赖于证据质量
- 投资人需要知道哪些是"事实",哪些是"推测"

#### 原则3: 分离线索与案例
```
signals/ (线索池,允许噪音)
    → 晋级标准: 有B级以上证据 + 值得深度研究
cases/ (案例库,高质量)
```

**为什么重要**:
- 不是每个AI应用都值得深度研究
- 快速登记(intake_card) → 观察3个月 → 决定是否晋级
- 避免案例库被低质量内容污染

### 3.1.1 扁平结构设计说明

**为什么用扁平结构?**
- 方便索引: 可以快速用id查询
- 方便统计: 计算各类失败模式出现频率
- 方便关联: related_modes字段建立失败模式网络
- 方便扩展: 新增失败模式不破坏现有结构

**如何使用?**

1. **在案例中引用**:
   ```yaml
   tags: ["failure:FM001", "failure:FM003"]
   ```

2. **在评估报告中引用**:
   ```markdown
   失败模式: FM001_流量红利消失
   严重程度: 高
   预警信号: CAC增长>50%
   ```

3. **未来扩展** (自动化):
   - 统计最常见的失败模式
   - 按赛道统计高发失败模式
   - 失败模式共现分析(哪些FM经常一起出现)

## 六、关键决策点

### 6.1 评分细则写进taxonomy/ (避免打分漂移)

**问题**: 不同时间/不同情绪,打分会漂移

**解决方案**: 创建 `taxonomy/scoring_rubric.yml`

```yaml
# taxonomy/scoring_rubric.yml
# 五维评分细则 v1.0
# 评分哲学: 档位化,避免伪精确

评分档位:
  红灯区 (0-3分): 有严重问题,不建议投资
  黄灯区 (4-6分): 有潜力但需验证,谨慎观察  
  绿灯区 (7-8分): 优质项目,可以投资
  强绿灯区 (9-10分): 顶级项目,强烈推荐

档内细分原则:
  - 档位是核心判断 (红/黄/绿/强绿)
  - 具体分数是档内细分 (如: 7.2分 vs 7.8分)
  - 不要过度纠结0.5分的差异
  - 关注"是否跨档" (如: 从黄灯到绿灯)

Unknown处理:
  - 如果某维度无法评分 → 标注unknown
  - 总分计算时,unknown维度不参与加权
  - 但要在报告中说明"缺少XX维度的评估"

用户需求 (0-10分):
  10分: 刚需+高频+有明确使用场景+用户愿意付费
  8-9分: 刚需但中频 或 高频但非刚需
  6-7分: 低频或场景模糊
  4-5分: 伪需求但有部分价值
  0-3分: 完全伪需求
  
  信息缺失时的处理:
    - 如果没有用户数据 → 标注"unknown (无用户数据)"
    - 如果没有付费验证 → 标注"unknown (尚未验证付费意愿)"
    - 如果场景不明确 → 标注"unknown (场景待验证)"
    - 不要: 因为"看起来应该还行"就打6-7分
    - 不要: 因为"项目早期"就一律打5分
    - 要: 诚实标注unknown,并在报告中说明需要验证什么

产品价值 (0-10分):
  10分: 创造新品类+10倍体验提升+有"啊哈时刻"
  8-9分: 显著提升(3-5倍)+用户愿意切换
  6-7分: 略有提升(1.5-2倍)+有差异化
  4-5分: "做得更好"但差异不明显
  0-3分: 无明显价值
  
  信息缺失时的处理:
    - 如果产品未完成 → 标注"unknown (产品未完成)"
    - 如果没有用户反馈 → 标注"unknown (无用户反馈)"
    - 不要: 强行打一个"看起来中等"的分数

技术壁垒 (0-10分):
  10分: 技术壁垒高+数据飞轮+大厂3年难复制
  8-9分: 技术有壁垒+有数据积累+大厂1年能复制
  6-7分: 技术壁垒中等+大厂半年能复制
  4-5分: 技术壁垒低+大厂3个月能复制
  0-3分: 无技术壁垒+任何人都能复制
  
  信息缺失时的处理:
    - 如果技术方案不明确 → 标注"unknown (技术方案待验证)"
    - 如果无数据积累 → 标注"unknown (数据飞轮未建立)"

商业模式 (0-10分):
  10分: LTV/CAC>5 + 边际成本趋近0 + 可规模化
  8-9分: LTV/CAC>3 + 边际成本低
  6-7分: LTV/CAC>2 + 商业模式清晰
  4-5分: LTV/CAC在1-2之间 + 盈利困难
  0-3分: LTV/CAC<1 + 单位经济不健康
  
  信息缺失时的处理:
    - 如果没有LTV/CAC数据 → 标注"unknown (无财务数据)"
    - 如果没有付费用户 → 标注"unknown (尚未商业化)"
    - 如果商业模式未验证 → 标注"unknown (D60前验证)"
    - 不要: 因为"看起来应该还行"就打6-7分
    - 不要: 因为"项目早期"就一律打5分
    - 要: 诚实标注unknown,并在报告中说明需要验证什么

团队执行 (0-10分):
  10分: 创始人完美适配+团队完整+执行速度快+有成功经验
  8-9分: 创始人适配+团队基本完整+执行正常
  6-7分: 创始人一般+团队有缺口+执行偏慢
  4-5分: 创始人不适配 或 团队不完整 或 执行慢
  0-3分: 多个维度都有严重问题
  
  信息缺失时的处理:
    - 如果团队信息不足 → 标注"unknown (团队信息不足)"
    - 如果无执行历史 → 标注"unknown (无执行历史验证)"
```

### 6.2 可证伪点设计 (强制字段)

**问题**: 很多项目说"我们会成功",但没有明确的验证标准

**解决方案**: 每个项目评估报告的YAML头必须包含:

```yaml
falsifiable_tests:
  - hypothesis: "转化率≥10%"
    test_period: "D60"
    kill_threshold: "<5%"
    scale_threshold: "≥15%"
    
  - hypothesis: "推理成本<$0.5/次"
    test_period: "D30"  
    kill_threshold: ">$1/次"
    scale_threshold: "<$0.3/次"
    
  - hypothesis: "用户完成率≥80%"
    test_period: "D30"
    kill_threshold: "<50%"
    scale_threshold: "≥90%"
```

**在报告中必须有**: Step8 可证伪点设计 (90天内如何验证核心假设)

## 九、关键注意事项

### 9.1 不要追求完美

**陷阱**: 想把所有失败模式都列出来,想把所有AI应用都研究一遍

**正确做法**:
- 先建立10个最常见的失败模式 → 边用边补充
- 先积累30个案例 → 够用了就开始用
- 先跑1个完整评估 → 发现问题再优化

**原则**: "Done is better than perfect"

### 9.2 失败案例比成功案例更重要

**为什么**:
- 成功案例容易陷入幸存者偏差
- 失败案例往往包含"预警信号"
- 创业者需要知道"什么不该做"

**行动**:
- 每积累2个成功案例,必须积累1个失败案例
- 主动搜索"AI startup failed"
- 建立 signals/deadpool.md (失败/关停/转型清单)

**对标缺口管理**:
- 每次评估如果缺少合适的对标案例,不要中断流程
- 将缺口记录到 signals/watchlist.md
- 每周集中补充2-3个缺口案例
- 3个月后缺口会越来越少(案例库成熟)

### 9.3 证据账本是生命线 (扩展)

**三个铁律**:

1. **没有链接/出处 = 不完整**
   - 每条证据必须可以复查
   - 3个月后你会忘记,链接是唯一线索
   - 建议: 重要证据截图存档

2. **可信度有标准,不是主观判断**
   - 参考 "2.4.2 可信度判定标准"
   - 证据等级 + 逻辑一致性 + 时效性
   - 多人使用时口径才能统一

3. **Unknown优于瞎编**
   - 宁可标注"unknown (无可靠数据)"
   - 也不要写"大约XX" (没有依据的推测)
   - 可以写"推算: XX (基于YY假设)" (但要标明推算逻辑)

**实操建议**:
- 用Notion/Airtable建证据库 (方便搜索)
- 重要证据用archive.org存档 (防止链接失效)
- 定期复查证据 (6个月一次,更新时效性)

### 9.4 index/ 目录的自动化策略

**问题**: 手工维护index.csv会成为负担,最终烂尾

**解决方案**: 三选一

**方案A: 暂不建立** (推荐Phase 1-2)
- Week 1-8: 不建index/目录
- 所有索引信息放在YAML头里
- 等案例库达到50+个时再考虑

**方案B: 提供脚本自动生成** (推荐Phase 3+)
```python
# scripts/generate_index.py
# 扫描cases/目录,从YAML头提取信息,生成index.csv
# 每周运行一次,或者git hook自动运行
```

**方案C: 使用数据库** (可选,高级)
- 用SQLite/Airtable替代CSV
- 每个案例md还是保留YAML头 (源头真理)
- 数据库作为"缓存"用于查询

**建议实施路径**:
- Phase 1-2 (Week 1-8): 方案A (不建index)
- Phase 3 (Week 9+): 方案B (提供脚本)
- 未来 (如果案例>500): 考虑方案C

**核心原则**: 
- YAML头是"源头真理" (单一数据源)
- index是"缓存" (可以重新生成)
- 不要手工维护两份数据

### 9.5 每周投入2-4小时维护

**时间分配**:
- 2小时: 研究1-2个新AI应用 (生成app_case_file)
- 1小时: 更新signals/watchlist.md (追踪线索)
- 1小时: 生成周报 (weekly_digest)

**长期复利**:
- Week 8: 30-50个案例
- Month 6: 200个案例
- Month 12: 500个案例
- **这将成为你最有价值的资产**

## 十、FAQ

### Q1: 我现有的44万字知识库怎么办?
